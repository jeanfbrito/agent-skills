# Commit

Create clean, well-organized git commits from the current working tree changes. The key idea: changes that serve different purposes get separate commits, because a reviewer reading the git log should be able to understand each commit independently.

## Workflow

### 1. Gather context

Use context-mode to run git commands that produce potentially long output. This keeps raw diffs and logs out of the main context window, saving tokens. Run a single batch call:

```
mcp__plugin_context-mode_context-mode__ctx_batch_execute(
  commands: [
    "git status",
    "git diff --stat",
    "git diff --cached --stat",
    "git log --oneline -10",
    "git log --oneline main..HEAD"
  ]
)
```

The `main..HEAD` log tells you whether this branch already has commits — this matters for the branch-name prefix rule below.

### 2. Analyze and group changes

Use context-mode to read the actual diffs — these can be very large and must not flood the context window:

```
mcp__plugin_context-mode_context-mode__ctx_batch_execute(
  commands: ["git diff", "git diff --cached"]
)
```

Then use `ctx_search` to query the indexed diff output for the information you need (e.g., what each file's changes do, which changes are related). This lets you reason about the diffs without pulling the full content into context.

Group files into commits by their **purpose**, not by file type or directory. Common groupings:

- **Linting/style fixes** (rubocop, eslint, prettier) — these are mechanical changes and should be their own commit
- **Test fixes or additions** — if they're fixing existing broken tests (not tests for new functionality), separate them
- **New feature code** — the core changes the branch is about
- **Refactoring** — structural changes that don't alter behavior
- **Configuration or dependency changes** — Gemfile, package.json, config files

Use your judgment. If a test file was added alongside the feature code it tests, they belong in the same commit — they serve the same purpose. But if someone fixed a flaky spec that's unrelated to the feature work, that's a separate commit.

When in doubt, fewer commits is better than too many. Don't split things that naturally belong together just because they're in different directories.

### 3. Create commits

For each group, stage the relevant files and commit. Use native git commands only — this ensures the user's configured git identity (name, email, GPG signing) is used correctly.

```bash
git add <file1> <file2> ...
git commit -m "<message>"
```

**Commit message rules:**

- Write a short subject line (under 72 chars) that describes *what* changed and *why*
- If the change needs more context, add a blank line and a body paragraph
- Use the imperative mood ("Add feature" not "Added feature")
- Do not include any AI attribution, co-author lines, or "Generated by" footers — ever
- Do not amend existing commits

**Branch name prefix** (e.g., `GSD-1234`): Include the ticket/branch identifier at the start of the commit message **only if this is the first commit on the branch** (i.e., `git log --oneline main..HEAD` shows zero commits before yours). This links the first commit to the ticket in issue trackers. Subsequent commits on the same branch don't need it — the branch name already provides that context.

Format when prefix is needed:
```
GSD-1234 Add legal document fetching from Lexion API
```

Format for subsequent commits:
```
Fix serializer to include nested document metadata
```

### 4. Generate MR/PR description

After all commits are made, produce a short summary suitable for a Merge Request or Pull Request description. To write this, look at **all commits on the branch** (not just the ones you created now) via context-mode:

```
mcp__plugin_context-mode_context-mode__ctx_batch_execute(
  commands: ["git log --oneline main..HEAD"]
)
```

The description should:

- Summarize what the branch accomplishes overall (1-2 sentences)
- List the key changes as bullet points
- Be written for a reviewer who hasn't seen the code yet — explain the *why*, not just the *what*
- Be concise — a few bullet points, not an essay

**Example output:**

> Adds backend support for fetching related documents from the Lexion API and displaying them in the legal dashboard.
>
> - Implement Lexion client method for related document lookup
> - Add `document_details` endpoint to surface related docs
> - Scope document visibility through `visible_legal_tasks`
> - Fix Lexion client stub in legal dashboard spec

Present this to the user so they can copy it directly into their MR/PR.

## Edge cases

- **Nothing to commit**: If `git status` shows a clean working tree, say so and stop.
- **Only staged changes**: If files are already staged, respect that staging — don't unstage and re-group. Commit what's staged, then handle any remaining unstaged changes.
- **Conflicts or merge state**: If the repo is in a merge/rebase state, warn the user and don't proceed with commits.
- **Untracked files**: Ask before including untracked files — they might be intentionally untracked (build artifacts, local config, etc.). Never add files that look like secrets (`.env`, credentials, keys).
